# 📅 개발일지

## 2026-03-16 (월요일)

### 오늘 한 일

- **Squad: 슬롯 기반 구조·세이브 적용 정리 (52f111e)**  
  - `Squad`를 고정 길이 슬롯 배열(`Character[4]`) 기반으로 리팩터링해서, 슬롯이 진실의 원천이 되도록 정리하고 `GetSlots`/`GetSlotOf`/`AddMember`/`ClearSlot`/`RemoveMember`/`SetSlots`/`SetPlayerCharacter` API를 재구성했다.  
  - `SquadController`에 `Initialize(CombatController)`와 `ApplySaveData(SquadSaveData)`를 분리해, 세이브 데이터 기반 슬롯 스폰·플레이어 결정·회전·동료 재배치를 한 곳에서 처리하고 `SquadSaveContributor`는 해당 API만 호출하도록 단순화했다.  
  - `SaveManager.LoadAsync`에서 저장이 없을 경우 `CreateDefaultSaveData`로 Celeste 한 명이 0번 슬롯에 있는 기본 세이브를 생성하도록 변경하고, `SquadSaveContributor.Gather`/`Apply`를 슬롯 기반 플로우에 맞게 정리했다.  
  - `PlayScene`과 `PlaySceneView`를 슬롯 배열(`IReadOnlyList<Character>`)과 `Squad.GetSlotOf(PlayerCharacter)` 기반으로 동작하도록 맞추고, 플레이어 변경/멤버 변경 이벤트에서 프로필 바인딩·선택 강조가 슬롯 인덱스를 기준으로 일관되게 동작하도록 점검했다.

### 기술 메모

- **슬롯 중심 설계**  
  - “슬롯이 참조를 가진다(Slot → Character)” 구조로 두면, 인벤토리/편성 화면/세이브 포맷을 모두 같은 개념으로 다룰 수 있어 설계가 단순해진다.  
  - 캐릭터가 SlotIndex를 들고 있게 만드는 것보다, `Squad.GetSlotOf(Character)`로 슬롯을 조회하는 쪽이 진실의 원천을 한 곳에 두기에 유리하다.

- **책임 분리**  
  - “누가 플레이어인지”에 대한 데이터 진실은 `Squad.Player`와 `Squad.SetPlayerCharacter`에 두고, `Character.IsPlayer`는 그 진실을 반영하는 파생 상태로 다루는 것이 구조적으로 깔끔하다.  
  - `Initialize`(컴포넌트 주입)와 `ApplySaveData`(세이브 적용)를 분리하면, `SquadSaveContributor`는 `ApplySaveData` 한 번만 호출하는 얇은 레이어가 되어 테스트와 유지보수가 쉬워진다.

- **세이브 기본값 전략**  
  - “세이브가 없으면 Controller 안에서 폴백 스폰”보다, `SaveManager`에서 디폴트 `SaveData`를 생성해 주고 이후 코드가 항상 `SaveData`의 존재를 신뢰하게 하는 쪽이 플로우를 단순화한다.

### 내일 할 일

- [ ] `PlaySceneView`/`CharacterProfileView` 정리  
  - `BindSquad`를 내부적으로 슬롯 단위 API(예: `SetCharacterProfileSlot`)로 분리해서, 추후 편성 화면이나 부분 슬롯 갱신에 재사용하기 쉽게 다듬기.  
  - 플레이어 변경/멤버 변경 이벤트에서 프로필 강조·바인딩 로직을 다시 한 번 점검하고, 불필요한 전체 리바인딩이 있는지 확인.

- [ ] 분대 편성 확장 준비  
  - 지금 슬롯 기반 설계를 기준으로, 향후 “스쿼드 편성 화면”에서 사용할 API 형태(`ClearSlot`, `AddMember(c, slotIndex)` 등)를 정리하고 TODO로 남겨두기.

- [ ] 세이브/로드 추가 검증  
  - 디폴트 SaveData(Celeste 한 명) 기준으로 실제 플레이 진입 → 저장 → 재실행 → 로드를 쭉 돌려 보며 슬롯/플레이어/HP/위치가 기대대로 복원되는지 확인.
